## 1. 問題箇所(severity 順)

### A. 実バグ — production UAF 級

#### A-1. [検証済] move の T_STRING: shared-root 文字列の buffer steal → 送信側 UAF
- 場所: `ractor.c:2534` 付近(`move_capture` T_STRING 枝)、`string.c:2675`(`str_dependent_p`)
- `rb_str_make_independent` は `STR_SHARED|STR_NOFREE` しか見ず、**STR_SHARED_ROOT の
  root は「independent」扱いで no-op**。root の `as.heap.ptr` を courier が奪い、
  materialize 後に free するため、同じ buffer を共有する子文字列が dangling になる。
- 再現(自明級):
  ```ruby
  a = "x" * 100
  b = a.dup                  # CoW: b が a の buffer を共有、a は STR_SHARED_ROOT
  port.send(a, move: true)   # a の buffer を steal → 受信後 free
  b[0]                       # UAF
  ```
- T_ARRAY 枝は `ARY_SHARED_ROOT_P` をまさにこの理由で除外しており、String 枝だけ
  対称性が欠けている。
- 修正案: shared-root(あるいは `!rb_str_reembeddable_p` 相当)は embed 経路
  (バイトコピー)へ落とす。A-4 / 改善 2(ゼロコピー方針)と合わせて決めるのが得策。

#### A-2. [検証済][複数一致 ×3] zombie thread wrapper unpin が main の local GC から foreign ページの shref_bits を非 atomic clear
- 場所: `thread_pthread.c:2467`(`rb_thread_sched_mark_zombies`)→
  `gc/default/default.c:7546`(`rb_gc_impl_unpin_in_flight_message`、plain `_CLEAR_IN_BITMAP`)
- `rb_thread_sched_mark_zombies` は `rb_vm_mark`(vm.c:3393)経由で **main の local GC
  でも走る**(gc.c:3423 の `global_gc || objspace == main` 分岐。no-barrier VM lock のみで
  worker は動いている)。worker 側の WB shref set / in-flight pin は plain `|=`
  (default.c:7461-7466, 7530-7532)。**同一 bitmap word 上の並行 RMW で lost-update**。
  worker の set が消えると shref pin を失った生きたオブジェクトが次の local GC で
  誤回収 = UAF(キュー上のメッセージ snapshot の pin 消失も同型)。
- コメントの正当化(「この reap は STW」thread_pthread.c:2454-2459, default.c:7538-7539)は
  main-local GC を STW と誤認。**zombie_threads_lock 撤去(2673868e1)が「shref の書き手は
  owner-only」という全 plain-store の前提を破った唯一の箇所**。design §1.4 が警告する
  「非 atomic の `bits[i] |= mask` は並行 set を片方消す」まさにその形。
- 修正候補: (a) reap(unpin + delist)を `rb_gc_during_global_gc_p()`(真の STW)時に
  限定し、main-local では mark のみ(最小)。(b) pin/unpin/WB set を atomic CAS 化。
  (c) unpin を owner Ractor に委譲。

#### A-3. [検証済][複数一致 ×2] Ractor 生成失敗時に `creating_child_objspace` が残置 → 二重列挙 / freed objspace 参照
- 場所: set = `thread.c:1096`、clear = `vm_insert_ractor`(ractor.c:653/680)のみ。
  raise 点 = `thread_create_core` の `rb_proc_isolate_bang`(thread.c:881 —
  外側変数捕捉の IsolationError、ごくありふれた失敗)・`rb_ractor_send_parameters`
  (thread.c:884)。どちらも `rb_ractor_living_threads_insert`(thread.c:926)より前。
- 失敗後:
  1. 死んだ子 Ractor が回収されると `ractor_free` → disown で zombie 帳簿に push され、
     `rb_gc_vm_each_objspace`(gc.c:4010-4011)が同一 objspace を **帳簿と
     creating_child の両方から二重列挙**(= b3b132728 で潰した二重 sweep と同じ壊れ方)。
  2. orphan merge が殻を free した後は creating_child_objspace が dangling になり、
     以後の全 objspace walk が freed memory を読む。
- **追跡中「rb_gc_vm_each_objspace lifecycle 不整合 UAF」(trace-verify 系)の具体的
  root-cause 候補として最有力**。
- 修正案: `rb_thread_create_ractor` に ensure を張り失敗パスでクリア
  (+ stillborn objspace の後始末方針を決める)。

#### A-4. [codex一致・双方critical] IO move が fptr 内の VALUE を無保護で運ぶ
- 場所: `ractor.c:2636-2645`(`MOVE_K_IO`)、courier mark = `ractor.c:2836` 付近
  (`u.io.klass` のみ mark)
- `struct rb_io` の `pathv` / `write_lock` / `timeout` / `tied_io_for_writing` /
  `encs.ecopts` / `writeconv_pre_ecopts` は送信側 objspace のオブジェクト。原本 husk 化後、
  送信側にこれらへの参照は無く、送信側の次の local GC / in-flight 中の global GC が回収
  → 受信側 IO が dangling VALUE を保持。`File` なら必ず `pathv` を持つので
  「simple IOs なら sound」というコメント(ractor.c:2637-2642)は成立しない。
  `io.path` / `inspect` で UAF。
- 修正案: fptr の VALUE メンバも courier ノード化(バイトコピー + 受信側で
  `RB_OBJ_WRITE` 再構築)し、husk 化前に source 側を null に。当面は T_FILE move を
  エラーにする選択肢もある(codex 提案)。

#### A-5. [検証済] unshareable→shareable 昇格の generic_fields 表移送順序 → 他 Ractor 読み手が rb_bug
- 場所: `ractor.c:1455-1467`(`rb_obj_set_shareable_no_assert`)→ `variable.c:2588`
  (`rb_mv_generic_ivar_to_shared`)、発火点 `variable.c:1377`
- `FL_SET_RAW(FL_SHAREABLE)` **してから** per-Ractor 表→global 表へ移送する。表の分類は
  `RB_OBJ_SHAREABLE_P` のみなので、フラグを観測した worker(例: 定数を rescue しながら
  ポーリング)が ivar を読むと global 表 miss →
  `rb_bug("Object is missing entry in generic_fields_tbl")`。
  コード中の「atomic である」コメントは GC safepoint の話で foreign reader には無力。
- 修正案: global 表へ insert(`generic_fields_lock` 下)→ FL_SHAREABLE → per-Ractor 表
  から delete、の順に(mutex acquire が挿入を可視化するので読み手は必ず見つける)。

#### A-6. [codex一致(2件)] 受信・move 経路の例外安全性クラスタ
- 場所: `ractor_sync.c:1112-1165`(`ractor_basket_value`)、`ractor.c:2500-2660`
  (courier build)、`ractor_sync.c:1027-1029`(`gen_fields_capturing`)
- 内訳:
  - `rb_marshal_load` はユーザの `marshal_load`/`_load` フックで raise し得るが cleanup
    が無く、`in_flight_materializing` スロットが残置 → containment verifier 恒久無効化、
    debug では次 receive の `VM_ASSERT`。**Ractor 終了後は zombie の `ractor_sync_mark`
    が freed snapshot を mark し続けて crash**(global GC の re-pin は zombie を
    スキップするため sender 側で回収済みになる)。
  - ユーザフック内の nested `Ractor.receive` が単一スロットを上書き→完了時 `Qfalse`
    リセットで**外側 snapshot の re-pin root が消え、global GC 跨ぎで load 途中の
    graph が回収** → UAF。
  - move courier build の raise(移動不可の子: `ractor.c:2648` の `rb_raise` 等)で
    courier・奪った buffer・fd・`seen` 表・basket が全 leak。husk 化済みの兄弟は
    **どこからも復元不能**(upstream move も部分 husk はするが、leak せず buffer は
    ヒープに残った — RLGC はデータ破壊 + 回収不能に後退)。codex はこれを
    「move の非トランザクション性」として critical 報告。
  - `gen_fields_capturing` フラグも raise で stuck(次 send の assert / stale map 混入)。
- 修正案: EC_PUSH_TAG/ensure 相当で「前値復元(スタック規律で nesting も解決)+
  courier/basket 解放 + capture 状態リセット」。`ractor_notify_exit` でスロットクリア。
  抜本的には preflight(無変異)/commit(no-raise)の 2 相化(codex 提案)。

#### A-7. objspace 併合中の finalizer 表 st_insert が GC を誘発し得る
- 場所: `gc/default/default.c:8708-8711`(`rlgc_absorb_finalizer_i`)、`8868`(呼び出し)
- st.c は `#define malloc ruby_xmalloc` なので dst 表 resize が malloc 会計を跨ぐと
  `garbage_collect_with_gvl` が走り得る。このとき src の finalizer 表は detach 済み
  (C ローカルのみが保持、誰も mark しない)で、**未移送分の finalizer proc が sweep
  → 併合後の表に dangling VALUE**。design §2.3 が明記する「併合の間は継承側の GC を
  禁止(`rb_gc_disable_no_rest`)」が外されている(コメントは alloc-free を主張するが
  st_insert が反例)。
- 修正案: 設計どおり merge 本体を GC 禁止で囲む、または finalizer 移送を非会計 alloc に。

### B. 実バグ — 条件付き / セマンティクス

#### B-1. `rb_gc_unregister_address` の cross-Ractor 呼び出しが silent no-op
- 場所: `gc.c:3892`、`ractor.c:348-359`
- 解除側は `GET_RACTOR()` のリストしか探さず miss は no-op。C 拡張は登録と解除を
  別 Ractor(継承スレッド・dfree 実行スレッド)で行い得る。解除失敗後、登録 Ractor の
  全 root scan が freed memory 上の `*addr` を `rb_gc_mark_maybe` — UAF read。
- 修正案: miss 時に VM lock 下で全 Ractor のリストを検索(解除は cold path)。

#### B-2. registered address の設計逸脱: local GC が自分のリストしか走査しない
- 場所: `gc.c:3400`、`ractor.c:423`
- design §2.1 手順 3.e は「アドレス登録スロットには後から**別 objspace の値**が入り得る
  から全リストを全 objspace が走査する」ことを根拠に per-objspace 分割を**不採用**と
  したのに、実装は per-Ractor 分割 + 自リストのみ走査。main の `Init_*` が登録した
  static スロットに worker が worker 生まれの値を書くと、**worker の local GC はどの
  リストからもそれを root にしない** → 誤回収。core 内の既知例(freeze_true_hash)は
  書き手=登録者に直してあるが、**汎用 C-API としての穴**が残る。
- 修正案: 全リスト走査を復活(foreign-skip が選別する、設計どおり)するか、
  「登録スロットへの store は登録 Ractor のみ」という C-API 契約変更を明文化。

#### B-3. [検証済] `rlgc.stalled_shareables` が global cycle でリセットされない → 恒久 global-GC 張り付き
- 場所: 書き込み `gc/default/default.c:6699`(gc_marks_finish)のみ、判定 `7845`、
  short-circuit `7891-7893`(gc_start 毎入場)
- 一度 `limit/2` を超えると、`gc_start` が **local mark を走らせず** `rlgc_global_gc` に
  short-circuit して戻るため、カウンタを更新できる唯一の場所に二度と到達しない。
  survivors 減 → limit 減で条件は保持され続け、その objspace の全 GC が STW global 化。
  メモリ安全ではないが自己回復不能な性能崩壊トラップ。
- 修正案: global cycle の step 5(`last_cycle_pinned = 0` の隣)で全 objspace の
  `stalled_shareables = 0`。

#### B-4. [codex一致] orphan merge が `zombie_total_pages` を減算しない
- 場所: `gc.c:4242-4260`(`objspace_absorb_disowned_zombies`)
- 手動 swap-remove しており `rb_gc_vm_forget_zombie`(gc.c:4147、減算つき)を使って
  いない。幻ページで trigger 3(default.c:7840)が余計な STW global GC を発火
  (cycle 末尾の再計測で自癒するが 1 回は走る)。
- 修正案: `rb_gc_vm_forget_zombie` を呼ぶだけ。

#### B-5. `rb_set_end_proc` の無同期 prepend vs barrier なし local GC の `rb_mark_end_proc`
- 場所: `eval_jump.c:59`、`gc.c:3411`
- publish store が初期化より先に見える(weak memory)と、並行 mark が未初期化
  `link->data` を `rb_gc_mark`(maybe ではない)。
- 「at_exit/END 非 main エラー化」(2026-06-30 決定)は**未実装**。実装後も
  main の prepend vs worker scan には release-store が必要。

#### B-6. cross-Ractor shareable walk が foreign objspace の `gc_rest()` を walker スレッドで実行
- 場所: `gc/default/default.c:3363/3400` 系(`objspace_each_exec` の `protected=TRUE`)
- barrier で止めた worker が lazy sweep 途中だと、TracePoint enable 等の walker が
  **worker の sweep(obj_free/dfree)を walker の Ractor identity で完走させる**。
  決定 16 の意図(local GC 中に任意の C 拡張コードを他 Ractor 文脈で走らせない)に
  反し、B-1 とも複合。
- 修正案: foreign は unprotected で歩く(shareable slot は local lazy sweep の対象では
  ない)か、pending sweep のある objspace をスキップして理由を文書化。

#### B-7. `dying_th->sched.finished = true` が unlock 後の plain store(ARM で危険)
- 場所: `thread_pthread.c:437`
- release 意味論は後続 store の前倒しを防がないので、reaper が `finished` を早く観測して
  `sched->lock_` 書き込み中の Ractor を free し得る(x86 は TSO で安全、ARM で露呈)。
- 修正案: atomic release store + `rb_thread_sched_mark_zombies` 側 acquire read。

#### B-8. move が String/Array/Hash のサブクラスを黙って落とす
- 場所: `ractor.c:2681-2694`(materialize)
- courier の MOVE_K_STRING/ARRAY/HASH ノードは klass を持たず plain クラスで再構築。
  upstream の move(slot memcpy)はサブクラスを保存していたので**互換性後退**。
  T_STRUCT の singleton も同様(T_OBJECT のみ再アタッチ)。
- 修正案: klass をノードに載せて allocate、singleton は copy 経路と同様に一貫処理。

#### B-9. `define_finalizer` の決定 12「shareable 一律拒否」が own-objspace shareable に効いていない
- 場所: `rb_gc_impl_define_finalizer`(判定は `GET_HEAP_OBJSPACE(obj) != objspace` のみ)
- 自分が作った Class/Module へは今も登録できてしまう。設計(一律拒否)と実装の
  どちらかに寄せる要決断。cross-objspace 側(拒否・copy no-op)は設計どおり。

#### B-10. 受信側 `ec->gen_fields_cache` に送信側 snapshot の dangling ポインタが残る
- 場所: `variable.c:1403-1410`、`ractor_sync.c:1127-1129`
- materialize 中の `rb_copy_generic_ivar` が受信 EC の cache に sender 所有の
  snapshot host / fields_obj を載せる。snapshot は送信側で回収され、**ページプール経由で
  同アドレスが受信側に再利用**されると `cache.obj == obj` が誤ヒットして foreign freed
  `fields_obj` を deref。
- 修正案: `gen_fields_materialize` クリア時に `ec->gen_fields_cache` を Qundef リセット。

#### B-11. join 経路の `rb_ractor_absorb_generic_fields` に GC-disable 規律が無い
- 場所: `variable.c:2624`(`gf_absorb_i`)、呼び出し `ractor_sync.c:958`
- 他の全 insert サイト(`rb_obj_set_fields` 等)が守る自己再入防止(resize xmalloc →
  自 GC の mark/sweep が同一表を触る)をここだけ欠く。GENERIC_FIELDS_PLAN.md の指定
  (VM-lock + gc-disabled 窓内)にも反する。
- 修正案: merge ループを `rb_gc_disable_no_rest()`/`rb_gc_enable()` で囲む。

#### B-12. modular GC(mmtk)ビルドの impl API 不整合
- 場所: `gc.c:775/812`(必須 dlsym: `rb_gc_impl_gc_rest` / `each_objects_shareable` —
  mmtk.c 未定義でロード失敗)。逆に `unpin_in_flight_message` / `during_global_gc_p` /
  `shref_marked_p` / `heap_page_count` / `objspace_absorb` はテーブル外で default 実装へ
  誤ルート(foreign heap のページヘッダを読む)。
- 修正案: テーブル + define + mmtk スタブを揃えるか、当面 modular GC をこのブランチで
  明示的に無効化。

#### B-13. pjob flush 例外パスが Ractor 宛てビットを global の triggered_bitset に再投入
- 場所: `vm_trace.c:2032`
- main 宛て orphan merge が任意 Ractor で走り得る(現状は absorb 先が
  `rb_gc_get_objspace()` なので single-writer は保たれるが、「main が併合する」設計から
  静かに逸脱。将来の per-Ractor pjob 利用者は壊れる)。
- 修正案: 残ビットは flush 中 Ractor 自身のマスクへ再投入。

#### B-14. fork 後の `zombie_threads` 残留 → dead Ractor / objspace が永久に併合されない leak
- fork 子では native thread が存在せず `finished` が永遠に publish されない。RLGC では
  各 global GC がその Ractor を mark し続ける(thread_pthread.c:2486)ため、objspace が
  永久に残る。upstream にも軽度の同型はあったが RLGC で悪化。
- 修正案: `rb_thread_atfork_internal` でリストを drain/クリア。

#### B-15. `r->sync.legacy` が mark されず、生存が dead main thread の `th->value`/`errinfo` への aliasing 頼み
- 場所: `ractor_sync.c`(`ractor_sync_mark` に legacy の mark なし)
- 例外パスは特に脆い: teardown 中に `th->ec->errinfo` を差し替える変更が入ると
  `Ractor#value` が freed object を raise する。zombie objspace は global GC に sweep
  される点が upstream と違う。
- 修正案: `ractor_sync_mark` の gated 枝(in_flight_materializing と同じゲート)で
  legacy を 1 slot mark する安価な硬化。

### C. CHECK/デバッグ限定・潜在

#### C-1. WB の foreign shareable 親 `a` への `RVALUE_OLD_P` TOCTOU(CHECK 限定)
- 場所: `gc/default/default.c:7380/7389`
- R2 が R1 所有の shareable `a` へ store するとき、R1 の major clear と競合して
  `gc_writebarrier_generational` の `if (!RVALUE_OLD_P(a)) rb_bug(...)` が CHECK ビルドで
  誤発火し得る。production は無害(b は shref が守り、remember は advisory)。
- 修正案: foreign 親は世代枝をスキップ(foreign 親の traversal はこの objspace の b に
  届かないので remember 自体不要)、または foreign `a` の assert を降格。

#### C-2. [codex一致・severity判断は相違] `rb_gc_single_objspace_p` が creating-child 窓を無視
- 場所: `gc.c:4318` 付近(codex 引用は gc.c:4199)、窓 = thread.c:1080-1097(swap)〜
  vm_insert_ractor
- Claude 側検証: 「in-flight pin(enqueue)〜cnt++ 間」は現状無割り当てで**潜在**。
  ただしこの無割り当て不変条件は未文書で、1 個の割り当て追加で pinned-roots スキップ +
  compaction 許可の安全性が崩れる。
- codex 側主張(critical 判定): 窓はより広く、`rb_proc_isolate_bang`(thread.c:881)/
  `rb_ractor_send_parameters`(thread.c:884)の割り当てが窓内で GC を誘発し得るのは事実。
  auto_compact が single 判定で main heap を動かすシナリオを提示。
- 突合の結論: codex の「窓内に割り当てが実在する」は正しい。ただし実害(移動)には
  「creator のマシンスタックから保守的に pin されないオブジェクトが子 objspace の
  wrapper からのみ参照される」ことが必要で、生成中の参照(isolated proc 等)は creator
  のスタック上 `params` に載っており pin される公算が高く、**実害の実証は未了**。
  双方の推奨修正は同一で修正コスト極小: **「creating 中は single 扱いしない」+
  debug assert(single ⇒ creating_child_objspace == NULL)を入れて塞ぐ**。

#### C-3. [複数一致 ×2][upstream 由来] `check_rvalue_consistency_force` の `while (empty_page)` が前進しない無限ループ
- 場所: `gc/default/default.c:1757-1768`(origin/master 1510-1521 と同一)
- `empty_page = empty_page->free_next` が欠落。CHECK ハントがこの診断経路に依存するので
  upstream へ修正を送る価値あり。

## 2. 改善提案

1. **会計系の pacing**: 単一 objspace 世界の local sweep で `shareable_objects` を
   減算しない → multi 移行直後に不要な global GC が 1 発。free 時 bit clear と同時に
   decrement、または single→multi 遷移で再計数。`stalled_shareables` も「pinned walk が
   mark した全数(shref の traverse 分を含む)」であり「新たに mark された shareable 数」
   (design §2.2)より過大 — 過剰トリガ要因。global recount が T_ZOMBIE を生存に数える
   軽微なずれもある。
2. **move の「ゼロコピー」が実際はコピー** [codex一致]: materialize は `rb_enc_str_new`
   (ractor.c:2685)でバイトコピーし、奪った buffer は courier_free(ractor.c:2787 付近)
   が捨てるだけ。受信側で buffer を adopt する(本当のゼロコピー)か steal をやめる
   (A-1 も同時に消える)か、方針を決める。現状は「危険だけ払ってコピーしている」。
3. **global sweep が作った他 objspace の zombie finalizer** は所有者の次の GC まで実行
   されない(quiescent worker で無期限遅延)— `rb_postponed_job_trigger_for_ractor` を
   owner 宛てに撃つ。
4. **objspace 殻の解放漏れ**: `rlgc_objspace_absorb` が `weak_references` darray・
   profile records を free せず、`malloc_increase` を捨てる(dead Ractor 単位の
   bounded leak)。
5. `rb_replace_generic_ivar`(variable.c:2472)は呼び手が消えた dead code — 削除推奨。
   現行の書き換えは「obj と clone が同 shareability class」という未文書の前提も持つ。
6. `newobj_init` に決定 13 の誕生時 assert(`shareable ⇒ wb_protected`)を追加すると
   規律が構造化される(writebarrier_unprotect 側の assert と対になる)。
7. FREEOBJ hook を worker objspace に立てない前提(design §3)が
   `rb_gc_impl_set_event_hook` で構造的に強制されていない — mask/assert で担保を。
8. moved T_OBJECT の singleton `attached_object` が送信側 husk を指し続ける
   (意味論の傷 + husk retention)— move での singleton 拒否か re-attach を。
9. **回帰テスト追加**(codex 提案、Claude 指摘群の再現にそのまま使える):
   - 途中で失敗する nested `move:` の原本保全(A-6)
   - moved IO + 受信前の強制 GC(A-4)
   - `GC.auto_compact = true` での初回 `Ractor.new`(C-2)
   - materialize 例外 → 続けてもう一度 receive(A-6)
   - これに Claude 分を追加: shared-root String の move(A-1)、
     Ractor.new 失敗 → 集中的 global GC(A-3)、昇格ポーリング(A-5)

## 5. 総評と優先順位

設計の骨格(封じ込め / single writer / mark-only shareable + shref / 帳簿による
列挙完全性)は実装とよく一致しており、正常経路の検証は厚い(WB 規律、in-flight pin の
正常経路、generic_fields 既知 4 バグの修正完全性、列挙モデルなどは clean 確認済み)。
指摘の重心は 2 パターンに集中:

1. **「owner-only / STW だから plain で良い」という論証の反例**(A-2 が代表。
   B-5, B-7 も同族)— ロック撤去シリーズの各コミットは個別に正しくても、撤去が
   重なると前提が交差で破れる。
2. **異常系(raise / 生成失敗 / 例外 teardown)の後始末漏れ**(A-3, A-6, B-11 が代表)
   — error path のレビュー・テストが系統的に薄い。

**着手推奨順**:

1. **A-1** — 再現自明・ユーザデータ破壊(String move)
2. **A-2** — 全 plain-store 設計の前提破れ(reap を STW 限定にするのが最小)
3. **A-3** — 既知 flaky UAF(trace-verify 系)の root-cause 候補。ensure 1 本で塞げる
4. **A-4〜A-7** — move/IO/昇格/併合の UAF 級
5. **C-2 / codex#3** — 述語硬化 + assert(安価)
6. **B 群** — B-3(性能トラップ)・B-4(1 行修正)は先に拾うと安い
7. **doc** — ロック撤去キャンペーンの反映(§2.1 3.e / §2.4 / mutex リスト)、
   決定 16【未実装】明記、id2ref 非互換の決定記録が上位

## 6. RLGC 自体の総合評価

### 結論: アーキテクチャは健全で、追求する価値がある

本レビューの約 25 件の指摘のうち、**設計の核を無効化するものは 1 件もない**。
「封じ込めが原理的に成立しない」「shref では守れないケースがある」「single writer が
構造的に破綻する」という類の発見は、6 エージェント + codex + 本体裏取りのどこからも
出ていない。バグは全て「設計の適用漏れ」であって「設計の欠陥」ではない。

### 設計の強み

- **2 つの約束(封じ込め / single writer)から全てを導出する構成**。バグ調査時に
  「どの前提が破れたか」を必ず特定できる形になっており、v1〜v2 の発見バグ史はすべて
  この語彙で説明が付く — 設計が診断言語として機能している。
- **mark-only shareable + shref** は v1 の教訓(shareable の traverse が foreign に
  踏み込む事故)を正面から潰した解。「shref の完全性 = 既存の世代別 WB と同じ規律」に
  帰着させた決定 13 は、新しい監査項目を増やさないという保守コストの観点で特に筋が良い。
- **終了系(zombie 帳簿・列挙完全性・value=併合してから返す)**も「特例を作らない」
  方向で一貫(3 経路とも「併合は継承者自身のスレッド」)。
- 性能の存在証明(8 Ractor 割り当て負荷で実効 ~7.7 コア)は出ており、
  「master は cache miss で直列化する」という動機に対する答えになっている。

### 弱点 — 指摘が集中した 2 つの構造的パターン

1. **「owner-only / STW だから plain store で良い」という論証の脆さ**。A-2 は
   ロック撤去コミットが単独では正しいのに、重なった結果 owner-only 前提が交差で
   破れた例。この種の論証はコメントにしか存在せず、**コミットを重ねるたびに黙って
   腐るリスク**を抱える。対策: plain store を「計測して正当化された最適化」に格下げし
   既定を atomic にする、または CHECK モードで書き手のスレッド所有権を assert する
   仕組みを入れる — 正しさの根拠を人間の帰納からコードの強制に移す。
2. **異常系の後始末が系統的に薄い**(A-3, A-6, B-11)。正常経路は
   btest/test-all/ASAN/TSan/CHECK で徹底検証されている一方、raise・生成失敗・
   例外 teardown のパスはレビューでもテストでも穴だった。設計の問題ではなく検証戦略の
   偏りで、「例外を投げながら Ractor を作る/送る/受ける」系の repro を rlgc_repro に
   足せば埋まる種類のもの。

### 固有のリスク(修正では消えないもの)

- **shref の WB 完全性は今後も単一障害点**。決定 13 は正しい緩和だが、「shareable への
  store を触る将来の全パッチ」が潜在的リスク源である事実は変わらない。CHECK verifier が
  この面を検査できているのは大きな資産で、維持すべき。
- **C 拡張エコシステムとの契約が最大の未解決フロント**。決定 16(T_DATA 宣言機構)は
  未実装のまま全 T_DATA が local GC に参加し、registered address の契約(B-2)も未確定。
  コア単体の正しさより、ここが upstream 化の律速になると予想される。
- compaction の multi-Ractor 無効化は現段階のトレードオフとして妥当だが、長寿命
  multi-Ractor プロセスの断片化として将来請求書が来る(設計に将来方針の記載あり)。

### サブシステム別の成熟度

| 領域 | 評価 |
|---|---|
| local GC 中核(pinned-roots / WB / sweep) | 高。clean 確認が厚く、残る指摘は A-2 の前提破れと会計系のみ |
| global GC・併合・帳簿 | 高。プロトコル自体は clean、指摘は会計とエラーパス |
