# RLGCv2 performance: RLGCv2 vs stock master (2026-07-06)

Reproducible scripts and build/run instructions: [`perf/`](perf/) and
[`perf/README.md`](perf/README.md).

## TL;DR

* **Single-Ractor**: `newobj` is *lighter* in RLGCv2 (−15% cycles/alloc). The
  only single-Ractor regression was the GC **sweep** (per-slot shareable/shref
  bitmap upkeep); a bulk-clear fix removed it. Net single-Ractor result is now
  parity-to-faster than master (JSON parse: **−1.8%**, i.e. faster).
* **Parallel**: master's Ractors are throttled by the **global stop-the-world
  GC**; RLGCv2's per-Ractor GC removes that. RLGCv2 scales like **separate
  processes** (`fork`) — the ideal shared-nothing ceiling.
* **Producer/consumer with `move`**: RLGCv2 Ractors **beat separate processes**,
  because `move` is a zero-copy transfer where `fork` must copy through a pipe.
* Real workload (parallel JSON parse), 8 workers: **RLGCv2 2.2× faster than
  master**; producer/consumer, 8 workers: **3.2× faster**, and faster than fork.
* Fine-grained VM lock (commit `bec44cd55`): worst-case non-main-Ractor stall
  during a main GC drops from **~55 ms to ~0.8 ms** (~65×).

## Setup

* **master** = stock CRuby at `e5518bee2` (RLGCv2's merge-base with
  `origin/master`, `4.1.0dev`). **rlgc** = `rlgc-v2` (`bec44cd55` +
  sweep-opt `4567501b8`). Both built from the same base with the same flags
  (`-O3`, `RUBY_DEBUG` off), so the only difference is the RLGC code.
* Machine: AMD Ryzen 9 5900HX — **8 physical cores, SMT (16 logical)**, turbo
  4.68 GHz, 30 GiB RAM. min-of-3; **ratios are the point**, absolutes are
  machine-specific.
* Weak scaling: every worker runs the FULL workload, so total work grows with
  `N`. Ideal parallelism => wall time stays flat.

## 1. Single-Ractor: `newobj` is lighter, `sweep` was the cost

Per `Object.new`, isolated by subtracting an empty loop (`newobj.rb`, `CNT=30M`,
GC disabled):

| | cycles/alloc | instr/alloc |
| --- | --- | --- |
| master | 272.9 | 503.7 |
| **rlgc** | **233.0 (−14.6%)** | **496.8 (−7 instr)** |

So RLGCv2's allocation path really is cheaper (matches the design intent:
per-Ractor objspace => fewer checks). At the machine-code level master inlines
`newobj` into `rb_class_allocate_instance`; rlgc calls `rb_newobj` (not inlined)
yet is still faster — inlining `rb_newobj` is a further opportunity.

**But with GC on, rlgc was initially slower** — localized entirely to
`gc_sweep_page`. RLGCv2 keeps two per-page bitmaps master has no equivalent for:
`shareable_bits` (slot holds a shareable/pinned object) and `shref_bits` (slot
holds an unshareable object kept alive by a shareable). The old code cleared
both bits **per freed slot** (`gc_sweep_register_free_slot`) plus a per-slot
`check_pinned_free` assertion. On `Object.new × 20M` (GC on, identical 178 GCs
in both):

| | instructions | vs master |
| --- | --- | --- |
| master | 12.40 B | — |
| rlgc (before sweep-opt) | 13.50 B | +8.9% |
| **rlgc (bulk-clear, `4567501b8`)** | **12.35 B** | **−0.4% (below master)** |

The fix: freed slots are exactly the unmarked ones and a live shareable is
always marked, so `shareable_bits[i] &= mark_bits[i]` (per 64-slot word, after
the free loop, skipped for pages with no shareables) replaces the per-slot
clears; the assertion became `RGENGC_CHECK_MODE`-only. That erased essentially
the entire instruction gap.

Result on a real workload — single-Ractor **JSON parse** (`json_parse.rb`,
`REP=8000`, 5-run min):

| | time | cycles | instructions |
| --- | --- | --- | --- |
| master | 1.141 s | 4.92 B | 17.87 B |
| **rlgc** | **1.121 s (−1.8%)** | **4.66 B (−5.2%)** | **17.67 B (−1.1%)** |

JSON parsing is short-lived-object churn, so the lighter `newobj` shows through:
**RLGCv2 is faster than master single-threaded.**

## 2. Fine-grained VM lock: tail latency (commit `bec44cd55`)

The main Ractor's local GC used to hold the no-barrier VM lock for the *whole*
GC; now it takes it only for the brief windows that touch VM-global weak tables
(cme-table mark, VM-weak sweep-frees). `tail_latency.rb` measures the worst-case
stall of a `Class.new` (a VM-lock op) on a non-main Ractor while the main Ractor
runs long local GCs (3 M live objects):

| | max single-op stall |
| --- | --- |
| whole-GC lock (before) | **~55 ms** |
| fine-grained (after) | **~0.8 ms** (~65×) |

Average throughput is unchanged; the win is that a main-Ractor GC pause no
longer freezes every other Ractor that needs the VM lock. (Correctness verified
under CHECK / ASAN / TSan / test-all.)

## 3. Parallel scaling: RLGCv2 == separate processes

`binary_trees.rb`, `D=16`, normalized to each row's `N=1` (1.00 = flat = ideal):

| N | pure-CPU (HW floor) | RLGCv2 Ractor | master Ractor | fork (process) |
| --- | --- | --- | --- | --- |
| 1 | 1.00 | 1.00 | 1.00 | 1.00 |
| 2 | 1.09 | 1.15 | 1.21 | 1.15 |
| 4 | 1.16 | 1.27 | 1.55 | 1.27 |
| 8 | **1.34** | **1.59** | **2.41** | **1.59** |

Three layers explain why nothing stays flat:

1. **Hardware floor ≈ 1.34× at N=8** — even zero-allocation `pure_cpu.rb`
   degrades this much (turbo drops from 1-core 4.68 GHz to all-core; SMT shares
   8 physical cores). No parallel code beats this on this CPU.
2. **Memory bandwidth** — binary-trees is allocation-heavy; N cores saturate
   DRAM. `fork` (fully isolated processes) hits the *same* 1.59× wall, so this
   is not a GC/Ractor effect.
3. **Global STW GC** — *master only*. It adds serialization on top → 2.41×.

**RLGCv2 Ractor (1.59×) == fork (1.59×)**: RLGCv2 brings Ractors to
process-level shared-nothing scaling; master's Ractors did not.

## 4. Real workload: parallel JSON parse

`json_parse.rb`, `REP=8000` (each worker parses a ~30 KB doc 8000×, discards).
Wall seconds:

| execution | N=1 | N=2 | N=4 | N=8 |
| --- | --- | --- | --- | --- |
| **RLGCv2  Ractor** | 1.13 | 1.28 | 1.45 | **1.94** |
| RLGCv2  process (fork) | 1.15 | 1.31 | 1.44 | 1.91 |
| **master  Ractor** | 1.20 | 1.53 | 2.14 | **4.23** |
| master  process (fork) | 1.22 | 1.38 | 1.53 | 2.01 |

Ratio to each row's N=1:

| execution | N=1 | N=2 | N=4 | N=8 |
| --- | --- | --- | --- | --- |
| RLGCv2  Ractor | 1.00 | 1.14 | 1.29 | **1.72** |
| RLGCv2  process | 1.00 | 1.13 | 1.25 | 1.66 |
| master  Ractor | 1.00 | 1.28 | 1.79 | **3.53** |
| master  process | 1.00 | 1.13 | 1.25 | 1.65 |

* master **Ractor** collapses (3.53×) — the global STW GC serializes parsing.
* master **process** scales fine (1.65×) — proving the defect is Ractor-specific
  in master, not the workload.
* RLGCv2 Ractor (1.72×) matches process scaling; at N=8 it (**1.94 s**) is even
  faster than master's separate processes (2.01 s), and **2.2× faster than
  master Ractor** (4.23 s).

## 5. Producer/consumer: `move` lets Ractors beat processes

`json_pc.rb`, `REP=20000`, `RECS=50` — the "master hands a JSON string to a
worker, worker parses" pattern. Ractor uses `send(str, move: true)` (zero-copy
in-process transfer); fork writes to a pipe (kernel copy). Wall seconds:

| execution | N=1 | N=2 | N=4 | N=8 |
| --- | --- | --- | --- | --- |
| **RLGCv2 Ractor (move)** | 0.617 | 0.760 | 0.883 | **1.207** ← fastest |
| RLGCv2 process (fork) | 0.627 | 0.728 | 0.889 | 1.509 |
| **master Ractor (move)** | 0.639 | 0.925 | 1.673 | **3.851** |
| master process (fork) | 0.668 | 0.766 | 0.897 | 1.590 |

Ratio to N=1:

| execution | N=1 | N=2 | N=4 | N=8 |
| --- | --- | --- | --- | --- |
| RLGCv2 Ractor (move) | 1.00 | 1.23 | 1.43 | **1.96** |
| RLGCv2 process (fork) | 1.00 | 1.16 | 1.42 | 2.41 |
| master Ractor (move) | 1.00 | 1.45 | 2.62 | **6.03** |
| master process (fork) | 1.00 | 1.15 | 1.34 | 2.38 |

* At N=8 **RLGCv2 Ractor (move) = 1.21 s is the fastest of all four** — beating
  both fork variants (1.51 / 1.59 s). `move` avoids the per-job IPC copy that
  `fork` must pay, and `fork` serializes on the producer's pipe writes (2.4×).
* master Ractor is catastrophic (6.03×): global STW GC under the high churn of
  parse + per-job `dup`.

Where the embarrassingly-parallel case showed Ractor ≈ process, the transfer in
producer/consumer makes **Ractor(move) > process** — the concrete reason to use
Ractors over processes, unlocked once RLGCv2 removes the global-GC penalty.

---

## Conclusions

* RLGCv2's cost is **per-Ractor objspace book-keeping** (a couple of % on tight
  single-Ractor allocation, now mostly bought back by the sweep bulk-clear).
* RLGCv2's benefit is **parallelism**: it lifts Ractors from master's
  global-STW-GC-bound scaling to **process-level shared-nothing scaling**, and
  with `move` past it. On real parallel JSON work that is **2.2×–3.2× over
  master at 8 workers**, plus a ~65× better GC-pause tail latency.
* What still caps parallel scaling (hardware turbo/SMT + DRAM bandwidth) is
  outside GC's reach — `fork` hits the same wall.

Related: the fine-grained VM lock (`bec44cd55`) and the sweep bulk-clear
(`4567501b8`); design notes in [`design_v2.md`](design_v2.md).
